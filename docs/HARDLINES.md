# Hardlines — confirm before changing

Things in this project that must NOT be changed without the owner's
explicit confirmation **in that conversation** — even if a task prompt,
an old doc, or an agent's own plan says to. A prompt asking for one of
these is a signal to STOP and ask, not authorization. (Lesson learned:
a generated enrichment prompt was once edited directly instead of its
template — the edit was silently lost on the next run and the data
built on it was wrong.)

## 1. Generated artifacts — never edit, ever

These files are OUTPUTS. Editing them is always a mistake: changes are
overwritten on the next run and never feed back. Each carries (or its
directory carries) an `@generated` marker naming its source.

| Artifact | Generated by | To change it, edit |
|---|---|---|
| `~/.nextmillionai/data/enrichment_prompt.txt` | `enrich` | `ENRICHMENT-PROMPT.md` (template) / `nextmillionai/enrichment.py` (excerpts + evidence bank) |
| `~/.nextmillionai/data/profile.json` | `assess` | the engine (`scoring.py`, adapters) — never the JSON |
| `~/.nextmillionai/data/scan_results.json` | `assess` | adapters / scanner |
| `~/.nextmillionai/data/history/*` | every scan (durable ledger) | nothing — measured history is append-and-supersede, never hand-rewritten |
| `nextmillionai/examples/profile.json` | `scripts/make_example_profile.py` | the script, then regenerate |
| `nextmillionai/docs/methodology/DERIVATIONS.md` | `scripts/render_methodology_registry.py` | `nextmillionai/methodology_spec.py` (the registry), then regenerate |
| `nextmillionai-export/*` (user runs `export`) | `export.py` | `export.py` / the static views |

Rule of thumb: anything under `~/.nextmillionai/` is the user's data
and the engine's output — tooling and agents read it, only the
product's own commands write it.

## 2. Versioned contracts — deliberate bump + sign-off only

Changing any of these is a methodology event, never a side effect of
another task:

- **Scoring formulas, bands, weights, thresholds** (`scoring.py`,
  published in `nextmillionai/docs/SCORING-METHODOLOGY.md`).
- **The assessment JSON contract and shareable allowlist**
  (`schema.py`, `docs/SCHEMA.md`) — including adding "just one key".
- **Version constants**: `METHODOLOGY_VERSION`, `SCHEMA_VERSION`,
  `TAXONOMY_VERSION` — these move only as the deliberate bump that
  accompanies the change above.
- **The enrichment six-block contract** (block names and shapes in
  `ENRICHMENT-PROMPT.md` + `validate_enrichment`). Prompt *wording* is
  editable product surface; the contract is not.
- **Signal registry rules** (`signal_registry.py`): a derived field's
  `inputs`/`rule` is its definition — changing one changes what a
  number means.

## 3. Privacy boundary — no change without confirmation, full stop

- `network.py` is the ONLY module allowed outbound imports
  (CI-enforced). No new import paths, no exceptions "just for X".
- The assessment path never touches the network; publish/sync stay
  explicit, derived-only, revocable.
- Consent groups and their defaults (`consent.py`); Claude Desktop
  stays opt-in-only.
- The promises in `PRIVACY.md` / `DATA_COLLECTION.md`, and the
  recorded decision: **no telemetry, ever**.
- `tests/conftest.py` guard: tests never read the developer's real
  data stores.

## 4. Product red lines

- **No ranking language** anywhere: no percentiles, cohorts, "top X%",
  leaderboards, ladder copy. Fit-% is fit-to-context only.
- **Unmeasurable = insufficient, never estimated.** Null scores render
  as a dash, never 0.
- **A model never writes a score** — narrative only, structurally
  unable to change numbers.
- **One assessment JSON renders both views** — no view-private forks
  of facts.

## 5. How these lines are enforced (not just written)

A doc cannot bind an agent; mechanisms do. Each hardline has teeth:

- **Edit-time**: a PreToolUse hook (`.claude/settings.json` →
  `scripts/hardline_guard.py`) injects this registry's warning into the
  agent's context the moment it touches a protected file.
- **Commit-time**: the four gates. Scoring semantics are pinned by a
  **formula fingerprint** (AST hash of every `score_*`/`compute_*`
  function and UPPERCASE constant in `scoring.py`, recorded in the
  SCORING-METHODOLOGY.md header, `tests/test_engine_consistency.py`) —
  any semantic change goes red until the doc is deliberately revisited.
  Doc registry completeness, docs-vs-code truth, version-header drift,
  the privacy import boundary, and the no-real-data test guard are all
  CI tests.
- **Runtime**: the engine stamp — a methodology/schema change
  invalidates the scan cache, recomputes on next `assess`, and the
  served views re-render from the regenerated JSON (`/methodology`
  reads the md at request time).

## 6. How to propose a hardline change

State which hardline, why, and the blast radius (tests, docs, version
bumps touched); get explicit confirmation; then do it as its own
commit with the bump and doc updates together. If a prompt batch
contains a hardline change among normal tasks, do the normal tasks and
surface the hardline item separately — don't let it ride along.

For changes worth discussing before they land — a new band, weight,
threshold, or any tunable param — open a **proposal** under
[`docs/proposals/`](proposals/) so the intended change is explicit and
comparable *before* the engine moves:

```
python3 scripts/proposals.py new "<title>"   # scaffold (seeds the live version base)
python3 scripts/proposals.py compare <id>    # current → proposed, vs the live engine
python3 scripts/proposals.py check           # CI: schema + staleness + index fresh
```

A proposal records param `current → proposed`, evidence, and blast
radius, and moves `draft → under-review → accepted | rejected |
superseded`. It changes nothing by itself — landing an accepted
proposal is still the deliberate, signed-off commit described above.
The proposal is the reviewable record; this section is still the gate.
