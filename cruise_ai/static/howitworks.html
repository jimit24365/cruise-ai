Quick start

One command runs the whole local pipeline — consent on first run, then scan, score, and open both views. Nothing leaves this machine.

$ python3 -m cruise_ai start # first run asks consent per source, then scans, scores, # and opens your Profile + Report on localhost
Want a look before consenting to anything? report --demo serves a bundled example profile and never touches your data.
start --codeadd the opt-in code scan: verified build domains, monorepo packages, test/CI footprint
start --rescanre-assess from scratch — after heavy coding days or tool updates
start --yesnon-interactive (CI/scripts); saved opt-ins are preserved

Command reference

Every command, in run order. Synopses show the invocation for this machine. [brackets] mark optional flags; all commands also accept --yes where noted, for scripts and CI.

python3 -m cruise_ai start [--code] [--rescan] [--project DIR] [--port N] [--no-open] [--yes]

The whole pipeline, one command: consent gate on the first run, then assess (cache-aware), then report — your browser opens on the finished profile. It deliberately stops at the privacy line: enrich, sync, and publish stay separate, explicit decisions.

expects
nothing — this is the one command a new user needs
runs
calibrate (first run) → assess → report
--codeInclude the opt-in local code scan in the assess step.
--rescanForce a full rescan instead of reusing the cache.
--project DIRScan a single project directory.
--port N · --no-openServer port (default 7749) · don't open the browser.
--yes, -yNon-interactive; saved opt-ins preserved.
python3 -m cruise_ai calibrate [--yes]

Onboarding and consent. Shows the full data-collection disclosure, then asks one question per source group (Claude Code, Cursor, Codex, Kiro, git, the wider tool field, local models, and the opt-in-only Claude Desktop). Your answers define the entire collection scope — nothing outside it is ever read.

expects
nothing — runs standalone, or automatically inside start
reads
no data sources yet (disclosure only)
writes
~/.cruise_ai/consent.json · collection_config.json
--yes, -yNon-interactive: accept defaults. Previously saved explicit opt-ins are preserved, never reset.
caveatsRe-running is safe and sticky: every question shows your current answer and keeps it unless you change it. Changing consent invalidates the scan cache — the next assess rescans with the new scope.
python3 -m cruise_ai assess [--rescan] [--code] [--project DIR] [--yes]

The engine. Scans every consented source, counts signals, supersedes totals from the durable ledger (measured history never regresses), and computes all scores as arithmetic against research-anchored bands. Writes the one assessment JSON that renders both the Profile and the Report. Ends with a coverage report of what was NOT collected.

expects
calibrate has been run (it will offer to run it if not)
reads
consented session stores · git logs · the durable ledger (~/.cruise_ai/data/history/)
writes
profile.json · scan_results.json · ledger entries
--rescanForce a full rescan, ignoring the scan cache. This is the re-assess command: run it after heavy coding days, tool updates, or whenever the staleness prompt suggests it.
--codeOpt-in local code scan: repo files reduced to metrics only (never stored, never sent). Adds verified build-domain verdicts, per-package monorepo classification, and test/CI/deploy footprint.
--project DIRScan a single project directory instead of everything discovered.
--yes, -yNon-interactive: accept defaults, preserve saved opt-ins.
caveatsThe cache is reused only when sources and engine version match — an engine upgrade or consent change triggers a rescan by itself. Durations are estimates (gap-measured active time where the tool records per-event timestamps, span-capped otherwise) and are always labeled as such. Unmeasurable signals render insufficient, never a guess. A nested git repo inside a project (vendored clone/submodule) is treated as its own unit — never counted into the parent.
python3 -m cruise_ai report [--demo] [--port N] [--no-open] [--yes]

Serves the Profile and the Report on localhost, both rendered from the one assessment JSON, and opens your browser. The two views flip into each other; tabs (Work, Lab, Provenance, Share) are identical on both.

expects
an assessment exists (run assess first) — except with --demo
reads
profile.json only
serves
/profile · /report · /methodology · /how-it-works
--demoServe the bundled example profile — never touches your data. Good for a first look before consenting to anything.
--port NServer port (default 7749). If busy, the next free port is picked automatically.
--no-openDon't open the browser automatically.
caveatsServing is local-only — this command never makes an outbound connection.
python3 -m cruise_ai sources [--yes]

Shows every data source discovered on this machine, what was collected from each, and at what fidelity (deep / counts / presence). The fastest way to answer "what is it actually reading?"

expects
nothing — read-only, safe anytime
reads
tool data directories (detection + counts)
writes
nothing
python3 -m cruise_ai enrich [--submit FILE] [--revoke] [--key KEY] [--yes]

Opt-in narrative on top of the numbers, written by your own agent — not by us, not by a server. Prints a prompt containing derived excerpts and an evidence bank of citable pointers (repos, commits, dated sessions — names and counts, never code). You run it through whatever agent you trust, then submit the result.

expects
an assessment exists
reads
profile.json + derived session metadata
writes
narrative blocks in profile.json (and only those)
--submit FILEValidate and merge your agent's JSON result. Rejected if it contains off-schema keys, ranking language, raw code, or anything score-shaped.
--revokeRemove the ingested narrative and restore the built-in heuristic text.
--key KEYDirect enrichment with your own API key, skipping the copy/paste loop.
caveatsStructurally cannot change a score — narrative fields are the only writable target. Every claim in the result must cite a pointer from the supplied evidence; submissions that invent pointers fail validation. The generated prompt file is marked @generated — edit the template, never the artifact.
python3 -m cruise_ai export [--out DIR]

Builds a static, self-hostable artifact: the redacted shareable JSON plus both views as plain files. Host it anywhere — no server of ours involved.

expects
an assessment exists
reads
profile.json
writes
./cruise_ai-export/ (or --out DIR)
caveatsThe artifact passes through the shareable-profile allowlist plus a verifier that refuses on private keys or filesystem paths. Lab-only and experimental signals never ship in an export.
python3 -m cruise_ai sync [--repo URL] [--status] [--revoke] [--yes]

One merged profile across your machines, through a private git repo you own. Each device pushes derived snapshots; every device merges deterministically (deduped by stable IDs). Scores are never merged — they are recomputed locally from the merged signals.

expects
a private repo you created (e.g. git@github.com:YOU/nma-sync.git)
sends
derived snapshots only — never sessions, code, or prompts
to
your repo. We are not in the path.
--repo URLYour private sync repo; stored after first use.
--statusShow sync state. Reads local files only — no network.
--revokeRemove this device from the sync store.
caveatsFirst use asks for explicit confirmation and shows what a snapshot contains. Keep the repo private; the tool warns, but the repo's visibility is yours to control.
python3 -m cruise_ai publish [--registry URL] [--dry-run] [--confirm]

The only step that shares anything beyond your own infrastructure: sends the curated, visibility-filtered shareable JSON to a registry — after showing you exactly what leaves. Fully revocable with unpublish.

expects
an assessment; a registry (defaults to a self-hosted one on localhost:7750)
sends
shareable JSON only (allowlist + verifier, same as export)
--dry-runPrint exactly what would be sent. Sends nothing.
--registry URLRegistry to publish to (default: http://localhost:7750, self-hosted).
--confirmSkip the interactive confirmation (for scripts).
caveatsRun --dry-run first — it is the contract. Publishing stores a revoke token locally so unpublish works without an account.
python3 -m cruise_ai unpublish

Revokes a publish: removes your profile from the registry using the locally stored revoke token.

python3 -m cruise_ai reset [--all] [--yes]

Deletes your local data and starts fresh. All state lives in ~/.cruise_ai/ (independent of where the repo is cloned), so a clean clone still shows your previous profile — this is the explicit way to forget it. Wipes profile, scan cache, consent, collection scope, visibility, enrichment, and the durable history ledger; --all also removes the self-hosted registry store, sync clone, and config. Confirmation required unless --yes.

caveatsLocal only — nothing is sent anywhere. If you published, run unpublish first; a local wipe can't reach the registry copy.
python3 -m cruise_ai network [serve|status] [--port N] [--host H] [--store DIR]

serve runs the reference registry — self-host your own publishing target. status shows your publish state from local files, without touching the network.

caveatsThis module is the ONLY part of the codebase allowed outbound network access, and that boundary is CI-enforced. The assessment path can never import it.
python3 -m cruise_ai guide

This map, in your terminal: every command, what it reads, what it produces, and where the privacy line sits.

python3 -m cruise_ai recommend [--category <cat>] [--json]

Get personalized, evidence-based coaching recommendations from your AI usage patterns. Categories: analytics, token_optimization, skills, project_memory, learning. All recommendations are private — never shared or exported.

python3 -m cruise_ai dashboard [--json]

Show your AI usage analytics: session counts, token estimates, cost breakdown by model, top projects, and tool distribution. Computed entirely from local data.

python3 -m cruise_ai teach [topic]

Step-by-step tutorials for AI productivity patterns. Topics: plan_mode, subagents, context_engineering, skills. Run without a topic to list all available tutorials.

python3 -m cruise_ai feedback [acted|dismissed|useful|not_useful] --action-type <type>

Provide feedback on recommendations. Records whether you acted on, dismissed, or found a recommendation useful. Used to calibrate future confidence scores and suppress dismissed suggestions. All feedback stored locally only.

In the Profile & Report

Both views render from the one assessment JSON; these are the controls and features inside them.

Profile ↔ Report flipboth views

The segmented toggle at the top flips between the Profile (the card you'd show someone) and the Report (the full document) — same data, two depths. The active tab rides the URL hash, so a deep link like /report#lab lands on the same tab on either surface.

The four tabs — Work · Lab · Provenance · Shareidentical on both

Work: your projects with language filters and per-repo detail. Lab: experimental, labeled, never-shareable readings (e.g. the solo-equivalent leverage estimate) plus growth edges. Provenance: every source that fed the assessment with its fidelity (deep / counts / presence), what was NOT collected, and the raw scan data. Share: everything about what leaves the machine — see below.

Right-click explainevery section

Right-click any section to get "how is this computed?" — a plain-language note of what the number measures and which of your signals fed it, with a jump to the exact formula on the methodology page. No number on either view is unexplained.

Ask your agentdimensions · lab

"Explore with your agent" buttons copy a prepared prompt that asks your own AI agent to dig into that dimension or Lab signal against your real local data — the same trust model as enrich: your agent, your machine, nothing sent anywhere.

Sharing controlsshare tab

Per-item visibility toggles decide what an export or publish may include; the preview shows exactly the redacted JSON that would leave. Download the shareable JSON or the profile card image directly, or copy the ready export / publish / unpublish commands. Lab and experimental readings are structurally excluded — no toggle can share them.

Print / Save PDF — Full or Snapshotboth views

Full prints the complete document with every explanation and dimension detail expanded; Snapshot is the one-page overview. Brand colors are kept in print. Use the browser's print dialog (the button on the Share tab) and choose "Save as PDF".

Common tasks

python3 -m cruise_ai assess --rescan

Re-assess — count new work. Plain assess reuses the scan cache when nothing changed; --rescan forces the full pipeline: re-scan every consented source, supersede totals from the durable ledger, and recompute every score. Run it after heavy coding days, after a tool update, or when the staleness prompt suggests it. Also available as start --rescan (re-assess + serve in one go). Measured history never regresses: a rescan can only confirm or raise ledger-backed totals, never lose them.

python3 -m cruise_ai report --demo

Try it without consenting to anything — serves the bundled, engine-generated example profile. Reads none of your data, writes nothing.

python3 -m cruise_ai enrich --revoke · sync --revoke · unpublish

Undo anything shared or merged. Every opt-in step has a revoke: enrich --revoke removes the narrative and restores the heuristic text; sync --revoke removes this device from your sync store; unpublish deletes your profile from the registry. Consent itself is edited any time with calibrate.

The privacy line

Drawn plainly: everything above the dashed arrows in the pipeline runs entirely on your machine with no network in the path (CI-enforced). The two dashed nodes — sync and publish — are the only outbound steps: both explicit, derived-only, revocable, and to destinations you control. start never crosses the line.
